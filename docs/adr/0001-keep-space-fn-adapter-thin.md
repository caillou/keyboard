# Keep the space-fn adapter thin; push decisions into the pure engine

The space-fn feature is split into a pure state machine (`space-fn-engine.lua`, unit-tested under busted) and a thin Hammerspoon adapter (`space-fn.lua`, glue around `hs.eventtap`/`hs.timer`). When logic that lives in the adapter turns out to need testing or hardening, the rule is to **move that decision into the engine** — where the existing spec suite covers it for free — rather than to mock the `hs` runtime and test the adapter in place.

We chose this over building an `hs` test shim deliberately, because the obvious move (mock `hs.eventtap`, test the adapter) is the wrong one here: the adapter's remaining glue fails obviously and instantly at runtime (e.g. a broken re-entrancy marker turns every synthetic event into chaos the moment space is pressed), and a fake `hs` only asserts against our *model* of the API, not the API — false confidence on the surface where reality matters most. The autorepeat-dedup fix (2026-05-30) is the template: a bug that surfaced in the adapter was resolved by making the engine's `pending` buffer idempotent, not by testing the adapter. The adapter is verified by using the keyboard; the engine is verified by spec. Keep it that way.
