# EmmyLua.spoon vendoring + autocomplete generation

## PRD

[2026-05-29-lua-dev-tooling](../prd.md)

## What to build

Vendor EmmyLua.spoon into the repo and wire it so Hammerspoon generates
EmmyLua annotation stubs (the Lua `.d.ts` analog) from the live installed `hs`
API. LuaLS later reads those stubs for `hs.*` autocomplete. Because the stubs
are generated from the installed Hammerspoon version they cannot drift from it.

- **Vendor** EmmyLua.spoon (committed) at `hammerspoon/Spoons/`, MIT header
  intact, with its upstream provenance (source URL and commit) recorded.
- **Setup symlink**: `script/setup` symlinks `~/.hammerspoon/Spoons/EmmyLua.spoon`
  to the vendored copy, creating `~/.hammerspoon/Spoons/` first if needed. The
  generator hardcodes its output to `hs.spoons.resourcePath("annotations")`; the
  symlink makes those writes land physically inside the repo at a real path,
  while the editor reads them at that real repo path (no symlink on the read side).
- **Generation trigger**: a guarded `pcall(hs.loadSpoon, 'EmmyLua')` in `init.lua`.
  The generator runs as a side effect of the Spoon's `:init()`; no separate
  `:start()`/`:generate()` call. A missing Spoon must be a no-op, not an error.
- **Reload filter**: make `reloadOnLua` ignore files under the EmmyLua
  `annotations/` directory, so generated `.lua` files inside the watched target
  do not trigger a Hammerspoon reload. Belt-and-suspenders against reload churn
  regardless of watcher load-ordering.
- **No custom version guard.** Rely on the generator's own mtime guard
  (`annotations/timestamps.json`): it regenerates only when Hammerspoon's docs
  JSON changes (i.e. on upgrade), so loading on every reload is cheap.
- **gitignore** the annotations directory
  (`hammerspoon/Spoons/EmmyLua.spoon/annotations/`) so generated output is never committed.
- **`make update-emmylua`** re-vendors the latest upstream generator, delegating
  to a small `script/` helper that encodes the canonical source URL and target
  path (the way `make install` delegates to `script/setup`).

## User stories covered

- 15 — stubs generated from the installed Hammerspoon version.
- 16 — stubs regenerate automatically on Hammerspoon upgrade.
- 17 — stub generation costs nothing on ordinary reloads.
- 18 — generating stubs does not trigger a Hammerspoon reload.
- 25 — generator vendored and committed (identical code per machine, offline-safe).
- 26 — `make update-emmylua` re-vendors the latest generator.
- 27 — generated annotation files gitignored.
- 33 — vendored Spoon retains MIT header and records upstream provenance.

## Acceptance criteria

- [ ] EmmyLua.spoon is committed under `hammerspoon/Spoons/` with MIT header and recorded source URL + commit.
- [ ] `script/setup` symlinks `~/.hammerspoon/Spoons/EmmyLua.spoon` to the vendored copy, creating the parent dir if needed; idempotent.
- [ ] `init.lua` loads the Spoon via a guarded `pcall(hs.loadSpoon, 'EmmyLua')`; a missing Spoon is a no-op.
- [ ] `reloadOnLua` filters out files under the `annotations/` directory; saving config does not trigger an extra reload from annotation generation.
- [ ] `hammerspoon/Spoons/EmmyLua.spoon/annotations/` is gitignored.
- [ ] `make update-emmylua` re-vendors the latest upstream Spoon via a `script/` helper.
- [ ] No custom Hammerspoon version guard is added.
