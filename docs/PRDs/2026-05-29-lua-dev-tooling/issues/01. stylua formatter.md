# StyLua formatter: install, config, make targets

## PRD

[2026-05-29-lua-dev-tooling](../prd.md)

## What to build

Add StyLua as the repo's one and only formatter, matched to the existing house
style so adopting it is effectively a no-op on the current tree. StyLua becomes
available both from the command line and (later, via the editor config issue) on
save.

- Install StyLua through the existing `brew bundle` flow by adding `brew "stylua"`
  to the Brewfile.
- Commit a `stylua.toml` at the repo root configured to the existing style:

  ```toml
  column_width = 100
  indent_type = "Spaces"
  indent_width = 2
  quote_style = "AutoPreferSingle"
  ```

- Add Makefile targets next to the existing `make test`:
  - `make fmt` — format the repo in place with StyLua.
  - `make fmt-check` — verify formatting without writing, non-zero exit on drift.

This issue does not run the one-time baseline reformat or commit its result —
that is the baseline-format issue, which depends on this one.

## User stories covered

- 2 — formatting matches the repo's existing style (2-space, single quotes, ~100 col).
- 5 — `make fmt` formats the whole repo from the command line.
- 6 — `make fmt-check` verifies formatting without modifying files.
- 22 — StyLua available through the existing `brew bundle` flow.
- 34 (partial) — formatter exposed as Makefile targets next to `make test`.

## Acceptance criteria

- [ ] `brew "stylua"` is in the Brewfile and `brew bundle` installs it.
- [ ] `stylua.toml` at the repo root contains exactly the four settings above.
- [ ] `make fmt` formats Lua files in place using StyLua.
- [ ] `make fmt-check` reports drift without modifying files and exits non-zero when the tree is not conformant.
- [ ] Running StyLua over the existing tree is a near-no-op (style already matches the config).
