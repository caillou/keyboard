# Build space-fn engine with spec suite

## PRD

[2026-05-29-space-fn](../prd.md)

## What to build

A pure-Lua state machine that decides, from a stream of abstract event tokens, what actions a space-fn adapter should perform. No `hs.*` imports — the engine must run under plain `lua` so its behavior can be verified by spec rather than by squinting at it.

The core rule is **release-order**: when space is held and another key is pressed, neither is emitted yet. The verdict comes at the first release event — if the other key releases first, the user chorded (emit the remap); if space releases first, the user rolled fingers (emit literal space, then the buffered key literally). One supporting 750ms long-hold timer covers the "held space, changed mind" case.

A second intent signal — the keymap itself — short-circuits the release wait: if the next key is *not* in the keymap, the engine aborts pending state immediately (literal space + literal key). This is what makes typing "and i" feel right without typing-burst heuristics.

The three states (`idle`, `pending`, `committed-fn`), the event vocabulary, the action vocabulary, and the transition table are specified in the PRD's "State machine" section. Implement them as written.

Tests live in `spec/space-fn-engine_spec.lua`, phrased as user-visible scenarios so they double as documentation. Roughly 13–15 `it` blocks covering: tap, roll, chord (single and multi-key), long-hold commit, pending unmapped abort, post-commit unmapped swallow, autorepeat suppression, and modifier propagation. The full list is in the PRD's "Scenarios covered" section.

- 16, 17 — unmapped abort, post-commit swallow

## Acceptance criteria

- [ ] `hammerspoon/space-fn-engine.lua` exists. No `hs.*` references.
- [ ] Public API: a constructor returning a stateful object with one `advance(event, payload?)` method that returns a list of action tokens.
- [ ] State machine matches the PRD's "State machine" section: three states, the long-hold timer, the buffer, the transitions exactly as written.
- [ ] Keymap is data, passed into the constructor (not hard-coded inside the engine).
- [ ] `spec/space-fn-engine_spec.lua` covers every scenario listed in the PRD's "Scenarios covered" section.
- [ ] `make test` runs green.
- [ ] No mocking framework needed — tests pass events in and assert on action-token lists.
