# Integrate space-fn with windows mode

## PRD

[2026-05-29-space-fn](../prd.md)

## What to build

Two changes that keep space-fn and the existing windows-layout modal out of each other's way, plus a one-handed-window-ops affordance that's been wanted independently.

**Disable hook.** While the windows-layout modal is active, space-fn must be off — otherwise pressing `space` inside the mode (which the mode already binds to "center window with full height, exit mode") could be intercepted by the engine, and chord keys like `space + h` could leak a synthetic `delete` into the app behind the modal. Compose the existing `entered` / `exited` callbacks (which manage the status-message overlay) with calls to `require('keyboard.space-fn').setEnabled(false / true)` so the status-message logic stays intact.

**Arrow bindings inside windows-mode.** Add `left / right / up / down` entries to `hammerspoon/windows-bindings-defaults.lua` that mirror the existing `i / j / k / l` actions. This serves a standalone purpose (one-handed window ops with just the arrow keys) and happens to align with what space-fn emits during chord-mode — though we still disable space-fn inside windows-mode anyway, because non-aligned chord keys (`h`, `u`, `o`, etc.) would otherwise leak.

## User stories covered

- 20 — spacebar still centers + exits windows-mode (no regression)
- 21 — chord keys pressed inside windows-mode do not leak synthetic events to the focused app
- 22 — arrow keys act the same as `i / j / k / l` inside windows-mode (one-handed window ops)

## Acceptance criteria

- [ ] `hammerspoon/windows.lua` calls `space-fn.setEnabled(false)` on modal enter and `setEnabled(true)` on modal exit, without breaking the status-message overlay.
- [ ] `hammerspoon/windows-bindings-defaults.lua` gains four entries: `{ {}, 'left', 'left' }`, `{ {}, 'right', 'right' }`, `{ {}, 'up', 'up' }`, `{ {}, 'down', 'down' }`.
- [ ] **Human verification:**
  - [ ] Enter windows-mode (Ctrl+s), press `space` → window centers with full height and mode exits, exactly as before.
  - [ ] Enter windows-mode, press `h` → window goes left-half (existing binding); no `delete` keystroke reaches the app behind.
  - [ ] Enter windows-mode, press `u`, `o` → existing window actions; no tab-switching in the focused app.
  - [ ] Enter windows-mode, press each arrow key → same window action as the corresponding letter (left=h-equivalent, etc.).
  - [ ] Exit windows-mode, type "and i" → space-fn behaves correctly again (re-enabled).
