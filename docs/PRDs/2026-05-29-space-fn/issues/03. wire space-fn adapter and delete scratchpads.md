# Wire space-fn adapter and delete scratchpads

## PRD

[2026-05-29-space-fn](../prd.md)

## What to build

The Hammerspoon-facing adapter that turns the engine into a working keyboard feature. The adapter owns one `hs.eventtap` filtered to keyDown/keyUp, translates real events into engine input tokens, executes the engine's returned action tokens against `hs.eventtap.keyStroke` and `hs.timer.doAfter`, and exposes `start()` / `stop()` / `setEnabled(bool)`.

The keymap lives at the top of the adapter as a plain Lua table — adding or changing a mapping should be a one-line edit:

```
i → up, j → left, k → down, l → right
h → delete
u → cmd+shift+[ (prev tab), o → cmd+shift+] (next tab)
v → ctrl+opt+v
```

Modifier propagation:
- Arrow/delete remaps emit the bare keycode and let macOS compose currently-held physical modifiers (so `cmd+shift+space+j` → `cmd+shift+left`).
- Multi-modifier remaps (`u`, `o`, `v`) emit their full prescribed combo.

Engagement gating: space-fn engages regardless of which physical modifiers are held at `space-down`. The decision to remap or not is the engine's job. Disabling is exposed only through `setEnabled(false)` (used later by windows-mode).

Re-entrancy guard: synthetic events posted by the adapter must not re-enter the eventtap callback. Set a flag before posting, clear after.

Logging: `hs.logger.new('space-fn', 'info')` at the top. `log.d(...)` at every state transition with from-state, to-state, event, and resulting actions. Off by default; flippable to `debug` from the Console.

Wire `require('keyboard.space-fn').start()` into `hammerspoon/init.lua` after the existing `require('keyboard.windows')` line.

Delete `hammerspoon/super.lua`, `hammerspoon/super-new.lua`, and `hammerspoon/super-backup.lua` in the same commit. Their content survives in git history.

## Post-implementation note (2026-05-30)

The "Re-entrancy guard" wording in this issue describes a Lua-level
`state.emittingSelf` flag set around synthetic emissions. The shipped
adapter in `hammerspoon/space-fn.lua` instead tags synthetic events
with `eventSourceUserData = SYNTHETIC_MARKER` and the tap callback
short-circuits on that property. Same goal, no flag-vs-async-post race
window.
