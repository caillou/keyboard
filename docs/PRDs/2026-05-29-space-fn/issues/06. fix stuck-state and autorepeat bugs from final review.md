# Fix stuck-state and autorepeat bugs from final review

## PRD

[2026-05-29-space-fn](../prd.md)

## What to build

The final-stage safety and right-thing reviews found three bugs that the per-issue reviews didn't catch because they only show at integration. All three share a common shape: state that survives a boundary it shouldn't, producing stray emissions into the focused app.

**Timer leak across the windows-mode disable boundary (M1).** When the user holds `space + h` and enters windows-mode mid-chord, the 750ms long-hold timer is still armed. On fire, the engine commits and the adapter posts a synthetic `delete` keystroke into the now-focused app behind the modal. Fix: in `space-fn.lua`'s `setEnabled(false)`, cancel the long-hold timer AND reset the engine instance (re-instantiate via `Engine.new({ keymap = KEYMAP })` so the next interaction starts from `idle`).

**Engine stuck in `pending` after re-enable (M2).** Same root cause as M1 — the engine retains pre-disable state across the gate, so the next `space` keydown after re-enable can find the machine in `pending` or `chord` with stale buffer contents. The reset-on-disable fix above kills both bugs at once.

**Autorepeat buffer duplication during `pending` (Minor #1, observed in practice).** Holding a chord key long enough to trigger macOS autorepeat (typically 250-500ms) accumulates duplicate buffer entries because `space-fn-engine.lua:50-56` inserts unconditionally. On key-up or timer-fire, every buffered entry emits a remap, producing N left-arrows for one held `j`. Fix: in the engine's `key-down` handler in `pending`, only insert if the key is not already in the buffer. Add a spec test for autorepeat safety (multiple `key-down` events with the same key during `pending` produce one remap on key-up).

**Stale doc comment in `windows-bindings-defaults.lua` (Minor #2).** Lines 23-24 claim `left`/`right` mean "send window to the monitor on the left/right"; the new arrow bindings at lines 38-41 actually mirror `j/k/i/l` half-screen. Rewrite those comment lines to match reality (something like `left/down/up/right => same as j/k/i/l (one-handed window ops)`).
